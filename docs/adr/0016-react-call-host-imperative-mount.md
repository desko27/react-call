# `react-call/host` is a host-agnostic imperative mount for multi-preview environments

The library ships a new subpath export `react-call/host` whose single named export — `mount(element, options?)` — mounts a React element into a body-level `<div data-react-call-host>` via its own `ReactDOM.createRoot`. The intended use is for environments that render multiple isolated React subtrees in parallel for previewing (Storybook autodocs page, Ladle, Histoire, react-cosmos) where the natural decorator pattern would mount one `<Confirm />` per preview and trigger the multi-root call-time throw from [ADR-0001](0001-multiple-roots-detection-at-call-time.md). The consumer calls `mount(<><Confirm /></>, { wrapper })` once from their host's preview-annotations module (e.g. `.storybook/preview.tsx`); a single Root listens for the lifetime of the page, regardless of how many preview subtrees the host mounts.

We chose this shape because the friction the issue tracker has surfaced (`#48`) is structural — sandbox hosts render N decorators per page, each one independently — and not specific to any one tool's API. The implementation is ~30 lines of `createRoot` + idempotency, with no Storybook plugin API surface, no `@storybook/preview-api` peer dependency, and no code that knows what a "story" is. Naming the subpath after the host pattern (`/host`) rather than the dominant tool (`/storybook`) keeps the entry point honest about what it does, leaves room for a future Storybook-specific addon to claim `react-call/storybook` if one is ever justified, and avoids embarrassment if the same pattern needs to be invoked from Ladle or a docs site that renders side-by-side previews.

A new term **Host** is added to [CONTEXT.md](../../CONTEXT.md): the environment that renders multiple isolated React subtrees in parallel for previewing. The Host is where the single shared Root lives, above or beside the previewed subtrees. This term did not previously exist in the project's vocabulary because the library had no concept of "rendering Callables outside the consumer's app tree"; the new entry codifies that concept now that the helper exists to support it.

## Considered options

- **Re-open [ADR-0001](0001-multiple-roots-detection-at-call-time.md) and allow multiple Roots at the library level** (e.g. last-listener-wins, or per-context-scoped listeners). Rejected because the single-Root invariant continues to be the right default for the common case — a consumer accidentally rendering two Roots in a real app — and re-opening it would regression the diagnostic signal that catches that bug. The multi-host scenario is deliberately a sandbox concern; solving it at the host integration layer keeps the core's invariants intact. The "which Root receives the `call()`?" problem also has no good library-level answer when the call originates from a non-React caller (a domain handler, a `useEffect`, an event listener) — there is no tree ancestry to disambiguate.
- **Storybook-specific addon registered via `addons: ['react-call/storybook']` in `.storybook/main.ts`.** Higher initial gloss (zero-touch from `preview.tsx`) but rejected on three grounds: (a) passing the consumer's Callables and wrapper into the addon requires either a brittle file-convention (`./.storybook/react-call.tsx`) or a Storybook-specific `parameters` schema, neither clean; (b) it couples the helper to Storybook's preview-annotations API which has churned between Storybook 7→8→9 and is forecast to churn again; (c) it forecloses on supporting Ladle/Histoire/cosmos with the same module, since each would need its own addon. The imperative helper is what `addons:` would have to call internally anyway — exposing it directly cuts out the middle layer.
- **Top-level export `mount` from `react-call`** (no subpath). Simpler exports map, but the helper imports `react-dom/client` and the core today imports only `react`. Adding `react-dom/client` to the main entry's import graph creates a footgun for SSR consumers and any bundler less rigorous about tree-shaking side-effect-free modules than Vite/Rollup. The subpath isolates the new peer dependency at the import-path level, mirroring the precedent set by [ADR-0012](0012-vite-plugin-displayname-injection.md) for `react-call/vite`.
- **Subpath named `react-call/storybook`** (the obvious tool-named option, originally floated in `#48`). Rejected: the implementation contains zero Storybook-specific code, so the name would lie. The asymmetry with `react-call/vite` is intentional — that plugin genuinely uses Vite-specific APIs (`parseSync`, the plugin lifecycle, `config.command`), so the name reflects implementation reality; this helper's implementation reality is "imperative React DOM mount," which is host-agnostic, so the name reflects that. A future addon that does Storybook-specific work (panel integration, parameter-aware mounting, lifecycle hooks tied to story navigation) can claim `react-call/storybook` then; burning it on a generic helper now is a hard-to-reverse mistake.
- **`react-call/sandbox`, `react-call/mount`, `react-call/preview` as alternative neutral names.** `sandbox` is overloaded with browser/iframe/security concepts and confuses with CodeSandbox. `mount` describes the action but not the use case, making it opaque to a consumer searching for "react-call with Storybook." `preview` leans toward Storybook semantically (Storybook's own iframe is called the "preview") and would feel oddly Storybook-flavoured for a host-agnostic export. `/host` is the most honest noun for the new vocabulary term.
- **Render via React Portal from inside each story's decorator instead of a second `createRoot`.** Pros: contexts from story decorators propagate naturally to the Confirm tree. Cons: every story preview still creates its own portal, which means N Roots active concurrently — back to the original problem. The only way to make portals work is to elect a single "owning" decorator that renders the portal and have all others abstain, which requires module-level coordination and depends on render order. The second-`createRoot` approach has a known limitation (decorator-derived React Context does not propagate) but is deterministic and predictable; the portal approach would replace a documented limitation with mysterious failures.
- **Bridge React Context between the story tree and the Confirm tree automatically** (something like `react-tunnel`). Rejected: the implementations of this pattern in the ecosystem are fragile, the consumer has to opt every Context into the bridge anyway, and the cleanest robust solution is for the consumer to use an external store (Zustand/Jotai/Redux/`useSyncExternalStore` over Storybook's `useGlobals`) which already crosses tree boundaries by construction. Library-side bridging would add complexity without removing the consumer's need to think about reactivity.
- **`mount(callables[], options)` (Shape B) or render-function (Shape C) or factory-builder (Shape D).** Considered and rejected. Shape B over-promises by typing the parameter as "an array of Callables" when internally the helper just renders a React element and doesn't care what it is — and breaks the public `RootProps` feature by removing the ability to pass props to the Root. Shape C mixes wrapper and content in a single function, harder to type and document. Shape D's factory is over-engineered for a use case that's literally "render this JSX into a div once." Shape A — `mount(element, options?)` — matches the workaround consumers already write by hand, with idempotency added.
- **Return a teardown handle (`unmount()`) from `mount()`.** Deferred. The use case is "call once at preview-annotations load time, live for the lifetime of the page." Exposing teardown invites consumers to call it from places it doesn't make sense (a story's `useEffect`, an addon's lifecycle), and the helper has no way to validate that. If a concrete use surfaces later, the `void` return can be widened to `() => void` without breaking existing callers.
- **Idempotency via DOM lookup (`document.querySelector('[data-react-call-host]')`) plus a `WeakMap` cache of containers→roots.** More robust than a `globalThis` symbol — survives even hypothetical HMR that reloads `react-dom/client` itself. Rejected as over-engineered: Vite does not currently HMR `react-dom/client`, the `globalThis[Symbol.for('react-call.host')]` approach is ~5 lines, and the DOM-lookup variant adds ~15 more lines for a scenario nobody has hit. Re-evaluable if the simpler approach fails in the wild.
- **Storybook canary app under `apps/storybook/`** mirroring the `apps/vite/` precedent from [ADR-0012](0012-vite-plugin-displayname-injection.md). Rejected for reasons that compound: (a) it contradicts the host-agnostic positioning by elevating Storybook to "reference host"; (b) each Storybook major requires migrating the canary, which is maintenance work the library pays for a benefit Storybook receives; (c) the integration is testable without Storybook — N concurrent subtrees in jsdom plus a single `mount()` is exactly the scenario, and `@testing-library/react` handles it natively. Unit tests in `packages/react-call/src/__tests__/host.test.tsx` cover the helper's correctness; if Storybook ships a major that breaks how preview-annotations or decorators work, that's a recipe-level documentation update, not a library bug.
- **Visual reference in `sites/demo`** showing N parallel previews + a `mount()` call. Considered as a low-cost middle ground between unit-only and a full canary. Rejected at the user's request: keeping `sites/demo` untouched holds the integration's footprint to a single package and avoids implying that the demo site is testing the helper. The README will reference unit tests for the test contract and show a Storybook-flavoured snippet that the user can copy verbatim.

## Consequences

- **New publishable subpath `react-call/host`.** `packages/react-call/package.json` `exports` gains `./host` with `import` / `require` / `types` entries mirroring the existing `./vite` shape. `typesVersions` gains a `host` entry for legacy resolver compatibility (same trick as `vite` and `mutation-flow`).
- **New optional peer dependency on `react-dom`** declared as `peerDependenciesMeta.react-dom.optional: true`, mirroring the existing optional `vite` peer. Consumers who use `react-call` without `react-call/host` (the vast majority — anyone with a real app rather than a sandbox) see no install prompt. React Native consumers continue to work with the core.
- **Build pipeline gains a third lib entry.** `packages/react-call/vite.config.ts`'s `lib.entry` map adds `host`. `rollupOptions.external` adds `react-dom/client` so it isn't bundled into the plugin's output (the consumer's own `react-dom` resolves it). New artifacts: `dist/host.{js,cjs,d.ts,d.cts}`. Tarball snapshot (ADR-0007) updates accordingly.
- **New `size-limit` budgets for `dist/host.{js,cjs}`.** Initial measurements: 336 B / 402 B brotli; budgets set at 400 B / 500 B with ~19% headroom (matching the relative headroom of the `mutation-flow` budgets). The helper itself is ~30 lines but the import statements (`createRoot` from `react-dom/client`, `jsx` from `react/jsx-runtime`) account for most of the bytes; `react-dom/client` is externalized so the consumer's own copy resolves it. The 1 KB budget on `dist/main.{js,cjs}` is unaffected; this is a separate budget on a separate artifact.
- **New term "Host" in [CONTEXT.md](../../CONTEXT.md)'s Core glossary.** Added inline during the design grilling rather than batched at the end of the ADR. The Relationships section gains one line establishing that a Host scopes the single-Root invariant. No existing terms change meaning; the addition is purely additive.
- **Reactivity contract documented in the README.** The wrapper can be reactive only if the source of reactivity does not depend on React Context inherited from the story's tree. Three patterns are documented as supported: `useState` / `useReducer` inside the wrapper itself, external stores (`useSyncExternalStore`, Zustand, Jotai, Redux), and Storybook's `useGlobals()` from `@storybook/preview-api`. The wrapper closure capturing a value at mount time is documented as the unsupported pattern, with the dark-theme example showing the supported alternative.
- **No changes to the core runtime.** `createCallable`, `Callable`, `Stack`, `Upsert`, the HMR registry, the displayName setter — all untouched. The 1 KB main bundle budget and its current measurement are unchanged. [ADR-0001](0001-multiple-roots-detection-at-call-time.md)'s reasoning is preserved verbatim; this ADR adds an out-of-tree mounting helper, not a runtime alternative.
- **Unit tests live at `packages/react-call/src/__tests__/host.test.tsx`.** The test simulates the autodocs scenario without Storybook: it renders N decorator-wrapped subtrees (each containing a `<Confirm />` that would, naively, be a Root), then calls `mount(<><Confirm /></>)` aside, then triggers `Confirm.call()` from a button inside one of the subtrees, and asserts the dialog appears exactly once with no `Multiple instances of <Root> found!` throw. This is the contract; the Storybook README snippet derives from it but isn't itself the test.
- **The README gains a new section: "Using react-call with a multi-preview host (Storybook, Ladle, Histoire, …)."** It shows the `mount` call from `.storybook/preview.tsx`, the wrapper pattern for static themes, and the `useGlobals` pattern for reactive themes. The section explicitly lists which hosts the helper has been verified against (Storybook, manually reproduced via the worktree experiment that drove this ADR) versus which are inferred to work by the host-agnostic design (Ladle, Histoire, cosmos).
- **Discoverability for Storybook users.** Search queries like "react-call storybook" resolve via the README section and issue tracker (`#48` already exists and surfaces). The subpath name `host` is not a discoverability liability because consumers do not search by subpath — they search by problem ("react-call with Storybook") and land in documentation that names `react-call/host` as the answer.
- **`react-call/storybook` subpath is intentionally reserved unused.** If a future addon wraps the Storybook addon API (panel integration, story-aware parameters, channel-based call triggering), that addon claims `react-call/storybook` and depends on `@storybook/preview-api` and `@storybook/manager-api` as proper peers. Until then, the subpath stays unallocated to avoid pre-committing a name to a feature that may never ship.
- **Failure mode on misuse: helpful, not silent.** If a consumer calls `mount()` from a server-rendered module (no `document`, no `react-dom/client`), the failure surfaces from `createRoot`'s own error rather than from a custom message — the helper does not wrap or normalize it because the underlying error already names the right culprit. If a consumer calls `mount()` twice with different `wrapper`s, the second wins and re-renders the same root with the new tree, no throw; this matches the spirit of "one Host per page" without making the second call a footgun.
