# `useMutationFlow` does not swallow throws from the user's `mutationFn`

The Trigger returned by `useMutationFlow` lets a rejected `mutationFn` propagate as an unhandled promise rejection. The competing direction was to keep the `.catch(() => {})` introduced with the hook in 2.0.0-next.3, which turned every throw into a silent no-op. We chose to remove it because the swallow operated entirely in the consumer's domain — deciding what to do with the consumer's errors — without affecting the lib's own state machine (`pending` lifecycle and in-flight guard, both handled by `.finally`). The swallow was sold in ADR-0014 and the README as the cause of "the dialog stays open on failure", but the dialog stays open because closing a Call requires an explicit `call.end()`, not because the throw was silenced; the original framing was wrong.

## Considered options

- **`.catch(e => console.error('[react-call] mutationFn threw:', e))` (visibility without `unhandledrejection`).** Restores telemetry via Sentry/Datadog's `console.error` integrations, keeps `unhandledrejection` quiet, doesn't break tests that depend on the throw not escaping. Rejected because it still picks a specific channel for the consumer's error; a consumer who wires telemetry through a different pipe gets a duplicate signal they have to filter, and the lib is still taking a decision in a domain it does not own.
- **`onError` opt-in as a third positional arg or options bag.** Gives the consumer an explicit seam without leaving the lib opinionated by default. Rejected because the seam already exists at the right level: a `try/catch` inside the consumer's own `mutationFn`. The `onError` callback is API surface for something the consumer can do trivially without it, and reintroduces the lib into the error-handling conversation we are explicitly trying to leave.
- **Status quo (`.catch(() => {})`).** Rejected. The swallow was not actually delivering the UX it was credited with — "stays open on failure" is a property of `call.end()` never running, not of the throw being silenced. What the swallow *was* delivering was active invisibility of the consumer's errors against `unhandledrejection`, the dev console, and any tooling hooked into either. That is the lib hiding domain it does not own.

## Consequences

- **Supersedes the "Throws are swallowed by the trigger" consequence in ADR-0014.** The rest of 0014 stands: hook-vs-primitive direction, the subpath entry, the narrow `MutationCall<Response>` view, the `.orEnd` chain for the Fallback response, the Manual-close path. Only the swallow consequence is reversed.
- **JSDoc on `MutationFn` in `packages/react-call/src/mutation-flow/index.ts` loses the swallow line.** The new body describes only what the lib owns: pending lifecycle, in-flight guard, narrow `call` argument. Errors are not mentioned.
- **The README's `# Async submission flow` paragraph is reframed.** The current wording "swallows throws so the dialog stays open" is replaced by an explanation grounded in the Stack / `call.end()` semantics: a `mutationFn` that doesn't reach `call.end` leaves the Call in the stack, the same way any Call without an `end()` would. The word "throw" does not appear; the lib does not promise anything about error handling.
- **`CONTEXT.md` is unchanged.** The MutationFlow and MutationFn entries were already neutral on error handling — they describe the lifecycle on throw as a property of the state machine, not as something the lib does *to* the throw.
- **Two tests in `packages/react-call/src/__tests__/mutation-flow.test.tsx` need an unhandled-rejection suppressor.** The test "thrown mutationFn keeps the dialog open and clears pending" and the first half of "Callable.update can swap the mutationFn between submits" intentionally throw to assert the UX contract. The assertions stay valid — what changes is that the rejection now escapes the trigger and would fail the test runner. Scoped to the relevant `describe`s via `beforeEach`/`afterEach` listeners on the test runtime's unhandled-rejection event; not applied globally, so unintended unhandled rejections in other tests still fail.
- **Bundle size goes down by a few bytes** in `dist/mutation-flow.js`; the existing `size-limit` budget gains margin. `dist/main.js` is unaffected — the hook ships from the `react-call/mutation-flow` subpath.
- **Not labeled BREAKING in the changeset.** `useMutationFlow` only exists from `2.0.0-next.3`; the stable 2.0 has not shipped. Early-adopter consumers on the `next` tag who rely on silent failures will notice a console warning at first run; the migration is "wrap your `mutationFn` body in `try/catch` if the failure matters to your UX or telemetry". Documented in the 2.0 changelog as a regular behavior note, not as a breaking change against the 1.x line (which never had the hook).
