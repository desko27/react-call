# Release flow and CI workflows

CI runs as two GitHub Actions workflows. `ci.yml` triggers on every pull request and every push to `main`, and runs six independent jobs in parallel — `lint`, `check-types`, `test`, `build`, `dist-lint`, `size` — each starting from a shared `.github/actions/setup` composite action that handles checkout, pnpm install, Node setup, and `pnpm install --frozen-lockfile`. `release.yml` triggers only on push to `main`, runs a single `release` job that builds + dist-lints + invokes `changesets/action@v1`, which either creates/updates a "Version Packages" pull request collecting unreleased changesets or — when the Version PR is merged — runs `pnpm release` to publish the new version to npm. PR concurrency cancels older runs of the same PR (so a force-push doesn't waste CI minutes finishing the prior run); release concurrency does not cancel (a publish interrupted halfway leaves npm in an undefined state).

We chose this shape because it mirrors the proven ffdev pattern (the same engineering team's other repo) — adopting it here means decisions about job graph, permissions, secrets, and concurrency don't have to be re-justified on every PR review. The release-specific deviations from a vanilla changesets setup are deliberate: building and dist-linting *before* the changesets/action call (rather than chaining them inside the `release` script) means a broken dist artefact fails the workflow before any version bump or publish happens, instead of after a successful PR merge. The build job in `ci.yml` and the build step inside `dist-lint` both build because they validate different things — the build job catches "does the lib build at all" early; the dist-lint job ensures the artefact it validates was rebuilt from the current commit, not cached from a previous run. The same applies to `size` — it builds, then measures, so a size regression cannot slip through because of a stale `dist/`.

`publishConfig.provenance: true` on `packages/react-call` opts the publish into npm's provenance attestation: the published tarball is signed with an OIDC-based statement linking it to the exact GitHub commit + workflow + run that produced it. Consumers (or auditors) can verify the link via `npm view react-call --json | jq '.dist.attestations'`. The `id-token: write` permission on the release job is what lets the runner request the OIDC token needed for this. Authentication for the actual publish also goes through OIDC via npm's "trusted publisher" flow — the repo + workflow file is registered as a trusted publisher in the npm package settings on npmjs.com, so the runner publishes without ever holding a long-lived `NPM_TOKEN`. No npm token secret exists in the repo; rotating credentials is no longer an operational concern.

## Considered options

- **One workflow with all jobs (no ci.yml/release.yml split).** Rejected: PRs would unnecessarily evaluate the release job (skipping it via `if:` works but adds noise to the PR's Checks tab and complicates the permissions model — release needs `contents: write` and `pull-requests: write`, which a default CI workflow should never request). Two workflows give clean separation: CI's permissions are read-only by default.
- **Per-job inline setup (no composite action).** Rejected: each job would duplicate the same 12-line stanza (`actions/checkout`, `pnpm/action-setup`, `actions/setup-node` with cache, `pnpm install`). Six jobs × 12 lines × inevitable drift over time is exactly what composite actions exist to prevent. Updating pnpm or Node versions is now a one-file edit.
- **`build` as a prerequisite to `dist-lint` (job-level `needs: build`).** Rejected because it serialises what should parallelise: the dist-lint job would wait for the build job to complete and then re-build anyway (jobs don't share filesystem state across workflow runs without explicit artifact upload, which is more machinery than the ~10 seconds saved). Each job rebuilds; total wall time is dominated by the slowest job (typically `dist-lint`), not by their sum.
- **`pnpm test` runs against `dist/` (consumer-fidelity).** Rejected per ADR-0006 (tests run against `src/` directly); the `dist-lint` job is the consumer-fidelity surface. Splitting them keeps the unit-test feedback loop fast.
- **Skip the `size` job; rely on local `pnpm size` invocations.** Rejected: a size budget that only fails locally is a size budget that fails after a PR has already been merged. The job adds ~30s to CI for a check that prevents the README's "Lightweight" claim from quietly eroding.
- **Use the `andresz1/size-limit-action` to comment size diffs on PRs.** Skipped for now (per Q15.d grilling). The `size` job already fails the PR on budget violations; the diff-comment bot is a nice-to-have that can be added later without changing the underlying workflow shape.
- **Use a long-lived `NPM_TOKEN` secret instead of trusted publisher.** Rejected: the trusted-publisher flow on npmjs.com pairs a specific GitHub repo + workflow file with publish rights for the package, removing the need to store, rotate, or audit a token. Configured once via the npm package's settings page; works with `id-token: write` + `provenance: true` already in place. The token-based alternative survives only as a fallback path for registries that don't yet support OIDC trusted publishers.
- **Run `pnpm release` directly in CI without `changesets/action`.** Rejected: `changesets/action` handles two responsibilities `pnpm release` does not — opening/updating the "Version Packages" PR that collects unreleased changesets, and deciding whether the current `main` push has a Version PR worth merging vs. has versioned files worth publishing. The action's logic distinguishes "user just merged changesets into main → open Version PR" from "user just merged Version PR into main → publish". Re-implementing that flow is a maintenance burden we'd own forever.

## Consequences

- **No npm authentication secret exists in the repo.** The release job authenticates via the OIDC trusted-publisher flow configured on npmjs.com (repo + workflow file pair registered as a trusted publisher for `react-call`). If the trusted-publisher config is ever removed, broken, or the package gets transferred to a different repo, the release job's publish step will fail with an npm auth error — at which point either restore the trusted-publisher entry or fall back to a short-lived granular `NPM_TOKEN` secret added to `env:`.
- **The first push to `main` after this lands will trigger `release.yml` even if there are no changesets.** changesets/action handles this gracefully (it simply exits with "no changesets found"), so no harm. Subsequent pushes with changeset `.md` files in `.changeset/` will produce Version PRs.
- **Prerelease publishing (e.g., `2.0.0-rc.0` on the `next` dist-tag) requires entering changesets pre mode** — `pnpm changeset pre enter next` on a branch, accumulating changesets, then version + publish from `main` while in pre mode. Exit pre mode with `pnpm changeset pre exit` before cutting the stable `2.0.0`. The decision to ship the 2.0 line as RCs on `next` (ADR-0003) becomes operational here.
- **No CI validation of these workflows is possible until a push to GitHub.** The YAML is syntactically valid (verified by reading) but semantic validation — does `actions/checkout@v4` exist, does the composite action resolve, does `changesets/action@v1` accept our inputs — only happens when GitHub Actions actually runs them. Expect a follow-up commit fixing whatever the first real run surfaces.
- **The legacy three-workflow setup (`bundle.yml`, `code.yml`, `tests.yml`) is removed in the same commit.** They referenced `npm ci` (we use pnpm), `node --run` script invocations (we removed the cross-package npm chains), and `npx playwright install chromium` (we dropped Playwright in ADR-0006). Keeping them around as broken-but-trigger-able would produce red CI badges on every push.
- **The `release` script in the root `package.json` is now just `changeset publish`.** Building before publishing is the workflow's responsibility, not the script's. A contributor who runs `pnpm release` locally with a stale `dist/` will publish a stale artefact — but this is not a supported local operation (release is a CI-only flow); the documented local actions are `pnpm changeset add` (record a changeset) and `pnpm bump-version` (run version locally to preview what would change).
