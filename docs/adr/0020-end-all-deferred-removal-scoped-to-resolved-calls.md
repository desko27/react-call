# End-all's deferred removal targets the calls it resolved, not the whole stack

`createEnd` captures the set of promises it resolves during its synchronous `set()` pass and defers the removal of *exactly those* (`store.remove(ending)`). Previously the deferred removal re-used the `null` promise as a "match everything" sentinel: `store.remove(null)` filtered the stack with `promise && c.promise !== promise`, which is falsy for every item, so it cleared the whole stack on the next macrotask. That status quo only worked because nothing was expected to mutate the stack between an end-all and its deferred flush. It broke the moment a `call()` (or `upsert()`) ran later in the same synchronous tick: `X.end(); X.call(...)` appended the new call, then the end-all's `setTimeout(() => remove(null), unmountingDelay)` wiped it along with the calls the `end()` had actually resolved. We scope the removal to the resolved set because that is what "end these calls, then unmount them after `unmountingDelay`" means — the unmount must apply to the calls that ended, not to whatever happens to be in the stack when the timer fires.

## Considered options

- **Snapshot the stack length or indices at end()-time and remove that prefix.** Rejected: positional identity is unstable. A targeted `end(promise)` or a `call()` interleaved before the timer fires shifts indices, so the removal would unmount the wrong calls. Promise identity is the stable key the rest of the store already uses (`set`, targeted `remove`).
- **Keep `remove(null)` as "clear all" but re-read the stack at flush time and only clear items already marked `ended`.** Rejected: a call added after the end-all is `ended: false`, so it would survive — but this conflates "ended by *this* `end()`" with "ended by anything". A second, targeted `end(otherPromise)` on a call appended after the end-all marks its own item `ended`; the blanket sweep would then unmount it on the first end-all's timer, collapsing two independent unmounting-delays into one. Per-`end()` capture keeps each removal independent.
- **Also drop the `null` sentinel from `set` and make end-all iterate explicit promises.** Rejected as out of scope: `set(null, ...)` is also used by `update()` (update-all) and is correct there — it mutates in place without scheduling a removal, so the same-tick hazard never existed for that path. Only the removal path needed the fix.

## Consequences

- **`store.remove` changes signature from `(promise: Promise | null)` to `(promises: Set<Promise>)`.** The `null`-means-everything branch is gone; the filter is now `!promises.has(c.promise)`. The store no longer has any "clear the whole stack" operation — clearing is expressed as "remove this set", and the empty/single/full cases all fall out of the same code. Targeted `end(promise, value)` passes a one-element set and behaves exactly as before.
- **`createEnd` allocates one `Set` per `end()` call** and populates it inside the existing `set()` updater (one `.add` per resolved call), so there is no extra stack traversal. `dist/main.js` stays within the 1 KB `size-limit` budget (810 B).
- **`update()` and targeted `end()` were never affected** and are now pinned by regression tests alongside the end-all case, in `packages/react-call/src/__tests__/end.test.tsx` under `same tick as later call()`. The tests flush the `unmountingDelay` macrotask before asserting, because the bug only manifested after the deferred removal ran — asserting synchronously would have passed even on the broken code.
- **`CONTEXT.md` is unchanged.** The Stack / Call / `end` vocabulary already describes this correctly — the bug was an implementation detail of removal scheduling, not a change to what those terms mean.
