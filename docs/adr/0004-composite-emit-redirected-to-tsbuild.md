# Composite typecheck emit redirected to `.tsbuild/`

`packages/react-call/tsconfig.json` sets `composite: true` (required for `tsc -b` to treat it as a referenced project) and `outDir: ".tsbuild"` — not `"dist"`. The `.tsbuild/` directory is gitignored and exists purely as a byproduct of typecheck-time emit; the canonical publishable artifact in `dist/` is owned by `vite-plugin-dts` (for `.d.ts`) and Vite's library build (for `.js` and `.cjs`).

We chose this split because composite mode forces TypeScript to emit `.d.ts` and `.js` files for the referenced project — there is no `composite: true` + `noEmit: true` combination TypeScript will accept — and `vite-plugin-dts` already owns `dist/main.d.ts` with the exact rollup shape we publish (`{ rollupTypes: true }` collapses the multi-file source into a single declaration the consumer sees). Letting `tsc -b` also emit into `dist/` would mean two tools racing for the same path: whichever ran last wins, and a stale `dist/` after a typecheck (with per-file `.d.ts` from tsc) would not match the rolled-up artifact a `pnpm build` produces. Redirecting `tsc`'s emit to a side directory removes the race entirely: `dist/` is always the vite output, `.tsbuild/` is always the tsc output, and neither tool needs awareness of the other.

## Considered options

- **Match ffdev's pattern: emit to `dist/` and accept the collision.** Their per-package `tsconfig.json` has `outDir: "dist"`, so `tsc -b` writes per-file `.d.ts` (and `.d.ts.map`) into `dist/`, then `vite build` overwrites with its rolled-up version. Functionally works in CI where order is fixed (build runs last), but locally a contributor who runs `pnpm check:types` then inspects `dist/` sees an artifact that isn't what `npm publish` would ship. Rejected: the cost of confusing dev state outweighs the cost of one extra gitignored directory.
- **Skip project references entirely; use `pnpm -r run check:types` with per-workspace `tsc --noEmit`.** Avoids the composite/emit problem because no workspace would be composite. Rejected for losing the things references unlock: incremental cross-workspace typecheck, automatic dependency ordering, and IDE go-to-definition across workspaces resolving through `references` instead of through stale `dist/*.d.ts`.
- **`composite: false` on `packages/react-call` + drop it from the root's `references` list.** Lets vite-plugin-dts be the sole owner of `dist/` without any tsc emit. Rejected: would leave `packages/react-call` outside the `tsc -b` graph entirely, meaning the root typecheck would skip it (defeating the point of having a single `pnpm check:types` that covers the monorepo).

## Consequences

- `.gitignore` includes `.tsbuild`. A `.tsbuild/` directory will appear under any composite package after the first `pnpm check:types`; it can be deleted at any time without losing meaningful state (TypeScript regenerates it).
- The invariant "`dist/` is what consumers see; `.tsbuild/` is what TypeScript sees" must be preserved as the build pipeline evolves. Any future change that wants composite emit to land in `dist/` needs to first remove `vite-plugin-dts` from the build to avoid reintroducing the race.
- Per-workspace `check:types` scripts use plain `tsc --noEmit`, not `tsc -b`. This stays valid because each workspace's `tsconfig.json` extends the base independently and can be checked in isolation. The root's `tsc -b` is the canonical "check the whole graph" command; per-workspace scripts are for iterating on a single workspace without rebuilding references.
- The `@typescript/native-preview` (tsgo) devDep is kept across workspaces but no longer appears in any `check:types` script. It remains available as `pnpm exec tsgo --noEmit` for ad-hoc fast typechecks while iterating; the canonical flow goes through `tsc` for stability with composite mode (which `tsgo`'s preview status does not yet guarantee).
