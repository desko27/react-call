# `Callable.displayName` is the only HMR registry key

The HMR persistence registry introduced in ADR-0009 stops reading `UserComponent.displayName ?? UserComponent.name` and instead keys exclusively off `Callable.displayName` — the property the consumer sets on the value returned by `createCallable()`, not on the UserComponent passed in. The lookup is wired through an `Object.defineProperty` setter on the Callable: assigning `Confirm.displayName = 'Confirm'` triggers, synchronously during module evaluation, either a registry lookup (adopt the previous store) or a registry insertion (this is the first evaluation). The closure's mutable store binding becomes a `storeRef.current` so that `call`/`upsert`/`end`/`update` track the adopted store transparently.

We chose this because ADR-0009's `UserComponent.name` path forced consumers into `export const Confirm = createCallable(function Confirm(...) { ... })` — a `const X = function X() {}` shape that no React developer reaches for naturally, that puts the name in two places at once (drift risk on rename), and that visually doubles the most important identifier in the call. The Callable-displayName path collapses both names into one: the natural form `export const Confirm = createCallable((props) => { ... })` becomes HMR-ready by appending `Confirm.displayName = 'Confirm'` — the same idiom React developers already know from `memo`, `forwardRef`, and HOC wrappers. The setter-on-assign timing is what makes the one-line addendum sufficient: by the time React renders the new Root (after Fast Refresh swaps the module reference), the registry adoption has already happened, so the first render sees the surviving store.

## Considered options

- **Keep both paths: `Callable.displayName ?? UserComponent.displayName ?? UserComponent.name`** (additive, no breakage for the existing `function Confirm(){}` idiom). Rejected because pre-2.0 is the only window to make this a single canonical form; coexistence preserves the very pattern this ADR exists to discourage and bifurcates documentation forever.
- **An explicit ID argument: `createCallable(arrow, { id: 'Confirm' })`** (the option ADR-0009 had rejected to keep the API surface). Rejected again here on the user's call: even though it eliminates double-naming, it expands the runtime API surface to encode something that is, conceptually, a property of the Callable — and `displayName` already exists in React for exactly this role.
- **Runtime introspection via `Error.stack`** to derive a stable ID from the call site. Rejected for the same reasons ADR-0009 rejected it: format varies across V8/JSC/SpiderMonkey, positions shift on edit, Vite/Rolldown rewrite source positions with HMR scaffolding.
- **Lazy registration at first Root render** (read `Callable.displayName` on mount, adopt from registry then). Rejected in favour of the setter trigger: with lazy registration the first render of the new Root sees a fresh store before adoption, requiring a mid-flight store swap and a re-subscription cycle through `useSyncExternalStore`. The setter path makes adoption deterministic and pre-render.
- **`Object.assign(createCallable(arrow), { displayName: 'Confirm' })`** as a one-statement form. Functionally equivalent to the two-line form but visually noisier and less React-idiomatic; not blocked, just not the documented shape.
- **A bundler plugin (`vite-plugin-react-call`)** that statically rewrites `const X = createCallable(arrow)` to inject the name. Deferred, not rejected — to be revisited as opt-in sugar on top of this decision once the runtime API is settled. A separate ADR will record that decision if/when it ships.

## Consequences

- **Breaking change for HMR persistence.** Consumers who adopted the `function Confirm(){}` workaround from ADR-0009 lose persistence until they add `Confirm.displayName = 'Confirm'`. Fast Refresh acceptance itself is unaffected — only the "open dialog survives the save" property regresses for un-migrated code. Pre-2.0 makes this acceptable; the public 1.x line never shipped the `function Confirm(){}` recommendation.
- **One canonical form.** The README's "Hot reload (HMR)" section collapses to a single recommended pattern; the `function Confirm(){}` diff disappears. The default `createCallable((props) => {...})` form stays minimal and the HMR opt-in is one extra line that names the thing once.
- **`Callable.displayName` is now load-bearing.** It is not cosmetic metadata; assigning a different value to it on a live Callable would shift its registry key. The setter therefore registers on first assignment only — subsequent assignments to the same value are idempotent, and to a different value are undefined behaviour (no-op in practice, no warning). Renaming a callable requires re-exporting; renaming `displayName` mid-flight is not a supported operation.
- **`UserComponent.displayName` / `UserComponent.name` become irrelevant to the registry.** React DevTools and the React Fast Refresh runtime still see them as before — naming the UserComponent for DevTools clarity is still encouraged, just no longer load-bearing for this library.
- **`storeRef.current` indirection.** `call`/`upsert`/`end`/`update` now resolve the store through a single-property ref instead of closing over `$store` directly. Negligible runtime cost; the indirection is the only thing that lets the adopted store replace the initial one without rewiring closures.
- **Collision behaviour is unchanged.** Two Callables in the same app with the same `displayName` still collide in the registry — namespacing by string only is the price of not introducing an explicit ID argument. The collision is now harder to trigger accidentally (you have to write the same string twice), and the diagnostic is identical to before ("dialog A and dialog B share state").
- **ADR-0009 is partially superseded.** The "HMR persistence is conditional on `UserComponent.displayName ?? UserComponent.name`" consequence and the rejected "explicit `hmrId` argument" option no longer reflect current behaviour; both are reopened and resolved here.
