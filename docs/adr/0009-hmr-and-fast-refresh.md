# Vite HMR + Fast Refresh: Callable as a component, store in a registry

> **Partially superseded by [ADR-0010](./0010-callable-displayname-as-hmr-key.md).** The registry mechanism, the `Object.assign(Root, {...})` shape, and the upsert-state move to the store all stand. The registry **key derivation** (`UserComponent.displayName ?? UserComponent.name`) and the rejection of the explicit-ID option are reopened and resolved in ADR-0010.


The object `createCallable()` returns is the `Root` component itself with the imperative methods (`call`, `upsert`, `end`, `update`, plus a `Root` self-reference for backwards compatibility) attached as properties via `Object.assign`. The store the Callable closes over no longer lives only in the createCallable closure — it is also registered in a module-level `Map` inside `react-call`'s own (HMR-immune) module, keyed by `UserComponent.displayName ?? UserComponent.name`. Together these two changes close GitHub issue #31: `vite-plugin-react`'s Fast Refresh accepts `Confirm.tsx` (no more "Could not Fast Refresh" warning, no full page reload), and for any UserComponent that has a name the open dialog survives across saves of its own source file.

We chose this shape because `vite-plugin-react`'s Fast Refresh refuses files whose exports are not "only React components" — and `createCallable()` previously returned the object literal `{ Root, call, upsert, end, update }`, which is not a component. Making the return value a function (the Root component) with extra properties is the smallest change that flips the export from "incompatible" to "compatible" while keeping `<Confirm.Root />` working unchanged (because `Confirm.Root === Confirm`) and adding `<Confirm />` as an equivalent JSX form. The registry is the second half: even with Fast Refresh accepting the boundary, Vite still re-evaluates the user's module body when the file changes, which would re-execute `createCallable()` and produce a fresh store with empty state — the dialog you were editing would still vanish. Lifting the store into the lib's own module keeps it alive across consumer-module re-evaluations; the new createCallable invocation finds the existing store by its `UserComponent.name` key and reuses it. The lib's module is itself loaded once per app session and never HMRs, so its `Map` is the right vault.

Two architectural side-effects make the refactor cleaner than the previous shape:

- The closure-owned `$upsertPromise` variable from ADR-0002 was unreachable through registry-reuse alone: the new createCallable closure has its own `$upsertPromise` binding, separate from the old one referenced by any in-flight upsert call's `end` callback. Moving that state into the store (a new pair of `getUpsertPromise` / `setUpsertPromise` methods plus reset on last-listener unsubscribe) makes the store fully self-contained, which is exactly what registry persistence requires. ADR-0002 picked "closure" over "store-owned" on the argument that `$upsertPromise` is not render state; the HMR requirement reverses the call. This ADR supersedes the closure choice for the upsert promise.
- The store's `onReset` callback (added in ADR-0002 to clear the closure-owned upsert promise) is no longer needed: the unsubscribe-when-empty cleanup now resets `upsertPromise` directly, alongside `nextKey` and `stack`. `createStackStore()` is back to a zero-argument factory.

## Considered options

- **Ship only the `Object.assign` change.** Gets Fast Refresh working (no full reload, no warning) but the open dialog still resets on every save because the user's module re-evaluation creates a new store. The original issue's headline complaint ("re-perform actions to open it to see my changes") would be only half-fixed. Rejected as half a solution where the second half costs the same architectural effort as the first.
- **Use a stable identifier derived from `Error.stack`** to support anonymous-arrow UserComponents without requiring a name. Rejected as fragile: stack frame format varies across V8/JSC/SpiderMonkey, source positions shift with code edits, and Vite/Rolldown wrap the user's code with HMR scaffolding that rewrites positions. The `Component.name` path is deterministic and under the consumer's control; documenting "give your UserComponent a name to get HMR persistence" is a one-sentence README addition with a soft-convention that React DevTools already rewards.
- **Add an explicit `hmrId` second argument to `createCallable()`** (or a third options-object slot). Rejected because the README brief was to keep the API surface. The current change is purely additive at the call site — existing code does not have to be touched — and the soft requirement "name your component" is a convention, not an API change.
- **Wrap the store creation in `import.meta.hot.data`**. Vite-specific, requires every consumer to opt in per their bundler, and only works for Vite. The registry approach is bundler-agnostic — Webpack, Rspack, esbuild, Bun all support Fast Refresh and all re-evaluate modules on update; the registry survives them all because it lives in `react-call`'s module, not the consumer's.
- **Keep the closure-owned `$upsertPromise` and add an "rebind" method on the store** for each new createCallable invocation to attach its closure's `upsertPromise`. Rejected as a layered version of the cleaner "store owns the state" model: every closure-managed piece would need its own rebind hook, and the store-owned model already passes 57/57 tests without any new API.

## Consequences

- **The public `Callable<P, R, RP>` type widens** from `{ Root; call; upsert; end; update }` to `FunctionComponent<RP> & { Root; call; upsert; end; update }`. This is additive — consumers who had `const x: Callable<...> = createCallable(...)` keep working. Consumers who constructed their own object literal typed as `Callable<...>` (rare) will now get a type error because their literal is not a function; the migration is to use `createCallable()` instead of hand-rolling, which is the only supported way to produce a Callable.
- **Two equivalent JSX forms now work**: `<Confirm.Root />` (preserved verbatim from 1.x; `Confirm.Root === Confirm` so the property lookup is identical) and `<Confirm />` (new, equivalent). Documentation can keep the `.Root` style if backwards-look-and-feel matters; both compile to the same code.
- **HMR persistence is conditional on `UserComponent.displayName ?? UserComponent.name`.** A function declaration (`function Confirm() {…}`) or a named arrow assigned to a `const` (which JavaScript names automatically) gives the registry a key; an inline anonymous arrow does not. Anonymous arrows still benefit from the Fast Refresh acceptance (no full reload, sibling state preserved) but the open dialog they own resets on save. The README's "Quick setup" example can be updated to use a named declaration to model the recommended pattern.
- **Two named UserComponents with the same name across the app collide in the registry**: the second `createCallable()` overwrites the first's store, and the two callables end up sharing state. Practical impact is small (developer notices immediately that "dialog A and dialog B are linked"), but the trade-off is real: namespacing by `name` only is the cost of not introducing an `hmrId` parameter. Users with naming collisions can disambiguate by renaming the component function.
- **Bundle size grew** from 719 B / 799 B (ESM/CJS brotli) to 841 B / 930 B — the registry + Object.assign + store-method additions added ~120 B. Still well within the 1 KB `size-limit` budget set in ADR-0007.
- **The isolation tests had to switch from named inline components (`const Component = ...`) to anonymous arrows via a factory helper.** `const`-assigned arrows inherit the variable's name (`Component.name === 'Component'`), which caused the two isolated callables in the test to collide in the registry — correct registry behaviour, but the opposite of what the test wants to exercise. Anonymous arrows have `name === ''` and so skip registration cleanly. The test file documents this intent inline.
