# Positioning lives on a dedicated `/why` page, not the landing or the README

The "why would I use this instead of rolling it myself?" argument gets its own top-level page at `sites/web` (`/why`, titled "Why react-call?"), distinct from `/concepts` (how the library works) and `/examples` (proof that it works). The landing stays a visual demo-reel and carries only a slim hook — a single before/after — that funnels to `/why`; it does not host the long-form argument. The README stays reference-only ([ADR-0017](0017-public-site-on-plain-astro-not-starlight.md)): positioning is not mirrored into it, so `/why` is the canonical home for it. The lead argument answers the real skeptic's objection — "why not just do this with React state, no library?" — for which a bare `useState(false)` boolean is the concrete on-ramp, not the whole of it. It runs on two prongs: (1) **co-location, rooted in an imperative/declarative mismatch** — a confirm/prompt/pick is imperative by nature (ask, wait, get an answer, continue), and expressing that through declarative React state is a shape mismatch, so the flow fragments (a resolver/callback channel, lifted state, the consequence stranded in a handler far from the intent). `await Confirm.call()` gives the *imperative* control flow an imperative API and keeps intent, question, answer and consequence at one callsite (the **Caller scope**), while rendering stays declarative in the Root — react-call goes with the grain of both rather than forcing one into the other. It does not make React less declarative; the Root still renders the stack declaratively (`useSyncExternalStore`). (2) **build-vs-buy** — the honest hand-rolled version (resolver, typing, SSR, the single-Root invariant, exit animations, a stack) converges on exactly what react-call already is, at ~1 KB with no deps. The page title is deliberately broad so further positioning arguments can land there without a rename.

## Considered options

- **An inline section on the landing.** Rejected: the landing's rhythm is "show, don't tell" (live demos, one-liners); a worked before/after + the hand-rolled-promise rebuttal + the concession is page-sized and breaks that rhythm. The discoverability cost of a page (a skeptic forms the objection in the first seconds and might never click through) is paid down by the slim landing hook instead, not by cramming the whole argument inline.
- **A page under `/concepts`.** Rejected: `/concepts` teaches mechanics (Root/Stack, Upsert, Mutation flow); positioning is a different job, and folding "why" into "how" muddies both.
- **Put the argument in the README.** Rejected: ADR-0017 makes the README the canonical *reference*, not a positioning surface. The site owns positioning; mirroring it into the README invites divergence.
- **Dedicated `/why` + slim landing hook (chosen).** The page holds the full argument; the landing catches the in-flow skeptic with the single strongest before/after and a link onward.

## Consequences

- **New top-level nav entry, "Why"**, alongside Examples and Concepts.
- **The site now has three positioning/learning tiers**: `/why` (positioning), `/concepts` (mechanics), `/examples` (proof). New "why X" content goes on `/why` rather than scattered into the landing or the concept pages.
- **The landing gains one before/after block** (the confirm case) plus a "Why → `/why`" link. The "you could hand-roll the promise — but that wrapper, generalised, *is* react-call" beat and the "for a pure show/hide toggle, React state (a boolean) is the right tool — no library needed" concession live on `/why`.
- **The README is unchanged** — positioning is not duplicated there.
- **`CONTEXT.md` is unchanged** — the argument leans on the existing **Caller scope** term ("the async logic and the response handling live here"); the hand-rolled boolean pattern is the foil, an implementation detail, and stays out of the glossary.
