# Dist-lint pipeline: publint, attw, bundle-content guard, tarball snapshot, size-limit

The publishable artifact in `packages/react-call/dist/` is validated by five independent layers, each catching a different class of mistake. `publint --strict` and `@arethetypeswrong/cli` (run via the `dist-lint` script) validate the structural shape of `package.json` and the dual-format type entrypoints. Two Vitest tests under the `dist` project (`bundle-content.test.ts`, `tarball-snapshot.test.ts`) validate the contents of the built bundles and the file list that ends up in the consumer's `node_modules/react-call`. `size-limit` (run via `pnpm size`) enforces a brotli-compressed budget on `dist/main.js` and `dist/main.cjs`. All five layers run against the same built artifact and can fail independently; the developer fixes whichever one points at a real problem.

We chose multi-layer validation because each vector requires a different shape of check. `publint` catches `exports` map misconfiguration but says nothing about what files are in the tarball. `attw` catches CJS/ESM type-resolution mismatches (the kind that produce `import` resolves to `.d.ts` but `require` doesn't kind of bug a dual-format package can ship without noticing) but doesn't inspect bundle contents. The `bundle-content.test.ts` implements the static React Native compatibility guard decided in the Q5 grilling — it scans `dist/main.js` and `dist/main.cjs` for forbidden symbols (`react-dom`, `document.`, `window.`, `navigator.`, `localStorage`, `sessionStorage`, `HTMLElement`, `Element.prototype`, `CSSStyleDeclaration`) and fails the moment any slip into a build, so an RN consumer never has to be the test environment. The `tarball-snapshot.test.ts` captures the list of files `npm pack` would publish (via `npm pack --dry-run --json`) into a checked-in `tarball.snapshot.txt`; the diff appears in PR review whenever it changes, forcing a deliberate decision before a leaked test file or sourcemap or stray declaration map ships to consumers. The same file also asserts the published `package.json` has no `preinstall` / `install` / `postinstall` scripts (closing the supply-chain attack vector that compromised npm tokens love) and no `bin` field (react-call is a library, not a CLI). `size-limit` keeps the lib's "Lightweight" claim in the README enforceable: any change that pushes a bundle past the 1 KB brotli budget fails CI instead of slowly bloating across versions.

## Considered options

- **`publint --strict` alone.** Validates structural correctness of `package.json` but inspects neither tarball contents, bundle code, nor type resolution scenarios. Rejected as insufficient — the dual-format types bug it surfaced during implementation (CJS condition pointing at `.d.ts` instead of `.d.cts`) was only the first of three real issues this commit fixes, and the bundle-content / tarball / size dimensions need their own checks.
- **Skip the bundle-content guard; rely on code review.** The original tooling brief documented "✓ React Native" as a public claim in the README. A claim with zero automated enforcement is a claim that breaks silently the day a contributor adds an `import { createPortal } from 'react-dom'` to extend the lib in a PR. A 25-line test that scans the bundle is the cheapest way to keep the claim honest; the rejected ADR-0006-style "real RN test environment" would have cost an order of magnitude more for an order-of-magnitude smaller win.
- **Skip the tarball snapshot; use a denylist of glob patterns** (`**/*.test.*`, `**/*.map`, `**/.env*`, etc.). Rejected: only catches anticipated leak shapes. A snapshot fails on any unexpected file regardless of whether the pattern was foreseen, forcing a human review of every delta in the published surface.
- **Full `package.json` content snapshot.** Capture the entire published `package.json` (normalising `version`) and diff on every change. Rejected for ruinous signal-to-noise — every devDep bump, peerDep adjustment, or `exports` tweak would regenerate the snapshot, training reviewers to accept it blindly. The dangerous-scripts assertions in `tarball-snapshot.test.ts` close the specific fields we care about without that cost.
- **Keep `bundlesize` instead of `size-limit`.** Rejected: `bundlesize` pulls `iltorb` for brotli measurement, which is a deprecated native binding that fails to install on modern Node without a postinstall workaround. `size-limit` uses esbuild internally with built-in brotli support and no native compilation, and is maintained.
- **Per-package vitest config with `projects`** (the recommendation I originally proposed in Q11 grilling). The user picked the alternative — a root `vitest.config.ts` with `projects` — for forward extensibility when future packages need their own dist tests. The cost is a tiny additional level of indirection now (config at root not at the package); the benefit is one place to add a third project later instead of duplicating per-package configs.

## Consequences

- **Snapshot regeneration is a deliberate gesture.** Any commit that intentionally changes the published file list (adding a `LICENSE` file, splitting `dist/` into entries, removing a stale file) must run `pnpm test:dist -u` and commit the updated `packages/react-call/src/tarball.snapshot.txt` alongside. The diff appears in PR review as plain-text added/removed paths — designed to make leaks visible to the reviewer.
- **`fflate` is pinned to `0.8.2` via `pnpm-workspace.yaml#overrides`.** `attw` (via `@arethetypeswrong/core`) gunzips the packed tarball using `fflate`'s streaming API, with a workaround for `fflate` issue #207 hardcoded in `core/dist/createPackage.js`. The 0.8.3 release of `fflate` regressed that workaround — extracting any tarball returns an empty file list, and attw crashes with `Cannot read properties of undefined (reading 'filename')` for *every* package. Pinning to 0.8.2 restores attw functionality. The override should be revisitable when attw releases a version that handles fflate 0.8.3+ correctly (track upstream: https://github.com/arethetypeswrong/arethetypeswrong.github.io ).
- **The `vite-plugin-dts` build emits only `dist/main.d.ts`; a tiny inline plugin (`copy-dts-to-cts` in `packages/react-call/vite.config.ts`) copies it to `dist/main.d.cts` at `closeBundle`.** Both files contain identical TypeScript declarations — for this library the type information is the same whether resolved from `import` or `require` — but the `require` condition of the `exports` map (per `publint --strict`) must point at a `.cts` extension specifically. The copy plugin is the simplest portable way to satisfy publint without dragging in a second declaration-generation tool. If react-call ever ships ESM-only and CJS-only types that genuinely differ, the copy plugin becomes a real generator.
- **`packages/react-call` does not yet ship a `LICENSE` file in its published tarball.** This is a pre-existing oversight inherited from the v1.8.x publishing setup (verified by inspecting the published 1.8.2 tarball — no `LICENSE.txt` either), surfaced by the tarball-snapshot test which lists what's actually being published. The repo's `LICENSE.txt` lives at the monorepo root, not inside the package's directory, so npm pack doesn't include it. Fixing this means either copying `LICENSE.txt` into `packages/react-call/` during build or maintaining a duplicate file there. Out of scope for this commit; tracked as a follow-up before the 2.0 cut.
- **`bundle-content.test.ts`'s forbidden-symbol list is the canonical place to add new RN-incompatible patterns.** If a future class of API surfaces (e.g., `performance.now()` becomes problematic on some RN platform), the symbol is added there, the test fails immediately on the next CI run, and the fix is local. The list is meant to grow.
- **`size-limit` budgets are pinned in `packages/react-call/package.json#size-limit`** at 1 KB brotli per file. The current build is 727 B (ESM) / 803 B (CJS), so there is ~200-300 B of headroom for additions before the budget needs revisiting. Tightening to e.g. 850 B would be the conservative move once the 2.0 cut stabilises and feature additions slow.
