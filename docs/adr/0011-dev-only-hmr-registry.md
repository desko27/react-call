# HMR registry is gated behind `process.env.NODE_ENV !== 'production'`

The store registry from ADR-0010 — the module-level `Map` and the `Object.defineProperty` setter on `Callable.displayName` — is wrapped in `process.env.NODE_ENV !== 'production'` guards, and the `Map` declaration carries a `/* @__PURE__ */` annotation. In a consumer's production build, the bundler substitutes `NODE_ENV` to `"production"`, dead-code-eliminates the guarded block, and (thanks to `sideEffects: false` in `package.json` plus the `@__PURE__` annotation) drops the unreferenced `Map` declaration too. Measured savings on the published bundle as size-limit measures it (esbuild, brotli, minified): **−91 B ESM / −91 B CJS**, ~10 % of the current bundle and ~9 % of the 1 KB budget set in ADR-0007.

The registry exists exclusively to let `createCallable` re-evaluate during HMR and adopt the previously-registered store. In a non-HMR runtime (any production app, any SSR build, any test runner with `NODE_ENV='production'`), the module evaluates once, the setter fires once, the `Map` gains a permanent entry that is never read again, and `storeRef.current` is never reassigned. Functionally the code is a no-op in production; carrying its bytes for a no-op when 90+ B is recoverable with a one-line idiom every React developer already knows is the wrong trade-off. The chosen idiom is exactly what React itself uses to gate its own dev warnings.

## Considered options

- **Dual export via `package.json` `exports` conditions** (`"development"` + `"production"` files, the React 19 shape). Rejected because it requires a doubled build pipeline, doubled `size-limit` budgets, doubled tarball snapshots, and a new `dist-lint` story — substantial new infrastructure for the same 91 B saving that one `if` already delivers.
- **`import.meta.env.DEV`** (Vite-style). Rejected as bundler-specific: Webpack, Metro, Rspack, and esbuild do not substitute it. `process.env.NODE_ENV` is the lowest common denominator across the entire React ecosystem.
- **`typeof __DEV__ !== 'undefined' && __DEV__`** (React Native style). Rejected because it only works where the bundler defines `__DEV__` as a global — Vite/Webpack/Rollup do not, leaving the guard always-false outside Metro.
- **Skip the guard, accept the 91 B**. Rejected on user prompt — pre-2.0 size-limit budget already shows the trade-off matters; "the registry is meaningful only in dev" is exactly the situation the React-idiom guard exists for.
- **Self-substitute `process.env.NODE_ENV` in the lib's own Vite build** (so the published file already encodes one variant). Rejected because it would force the lib to ship two artifacts (the dual-export path) or to commit to dev or prod at publish time, neither of which is desirable. Vite library mode already preserves the literal by default — the consumer's bundler is the right place to make the decision.

## Consequences

- **Consumer must use a bundler that substitutes `process.env.NODE_ENV`.** Vite, Webpack, Rollup, esbuild, Rspack, Metro, Parcel, and every framework built on them do. Native ES modules served directly in a browser do not — those consumers would download the dev-guarded code (functional, just slightly larger). React itself makes the same assumption, so anyone consuming React already has the substituting bundler.
- **The lib's own Vite build must not substitute `NODE_ENV`.** Verified: `vite build` in library mode leaves `process.env.NODE_ENV` as a literal in `dist/main.js` and `dist/main.cjs`. No `define` override needed in `vite.config.ts`. If the Vite default ever changes, the override `define: { 'process.env.NODE_ENV': 'process.env.NODE_ENV' }` keeps the literal intact.
- **`storeRef.current` indirection stays in production.** It is the mechanism that lets the dev setter adopt the existing store, and removing it from the prod path would require either two source variants or runtime branching on every method. The indirection itself costs only a handful of bytes; the bulk of the saving comes from eliminating the `Map` and the setter, both of which the guards already handle.
- **Tests run with `NODE_ENV='test'` or `'development'`.** The guard evaluates true, so the registry stays active during the unit and dist test suites. The isolation tests (which intentionally avoid setting `displayName` so each `createCallable` gets a fresh store) continue to behave identically.
- **`size-limit` measures the post-substitution size.** Its esbuild preset compiles with `NODE_ENV='production'` by default, which is how it can report 803 / 881 B brotli for a published bundle whose raw source contains the dev block. The reported number is what the consumer's prod build actually ships.
- **Forgetting the guard is a foot-gun.** Anyone adding new dev-only diagnostics in the future must remember to gate them the same way. The convention is documented inline in `createCallable/index.tsx` next to the existing guards.
