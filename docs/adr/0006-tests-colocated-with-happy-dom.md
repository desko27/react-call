# Tests colocated in `src/__tests__/` with happy-dom + @testing-library/react

Tests live inside the library package they exercise — `packages/react-call/src/__tests__/` — and run via Vitest against happy-dom, using `@testing-library/react` for rendering and queries, `@testing-library/user-event` for interactions, and `@testing-library/jest-dom` for DOM-aware matchers. The dedicated `packages/tests/` workspace (which consumed `react-call` through its `dist/` symlink and ran Vitest in browser mode against real headless Chromium via Playwright) is deleted; what was once a third workspace collapses into the library workspace it was always exclusively testing. Test files import the lib via relative paths (`../../createCallable`, `../../types.public`) rather than the package name, making it explicit which source they exercise.

We chose this because the original watch loop was broken at its foundation: edits to `packages/react-call/src/` had no effect on tests until `pnpm --filter react-call build` ran first, because the `tests` workspace resolved `react-call` through `node_modules/react-call/dist/`, not through src. Colocating the tests with their source removes the build step from the loop entirely — Vitest watches `src/**/*.test.{ts,tsx}` and re-runs affected tests on save with no intermediate compilation. The same change unlocks the secondary wins ffdev's pattern offers: happy-dom is roughly five times faster than headless Chromium for our shape of tests (~700ms cold start vs ~3s), CI no longer downloads ~90MB of browser binaries on every fresh install, and the monorepo loses a workspace whose only purpose was test isolation.

The real cost is fidelity to a real browser: happy-dom emulates the DOM, the Event constructors, and a Window-like global, but is not Chromium. CSS-driven behaviour (layout, computed styles, hit testing) and certain microtask/event-loop orderings can diverge from a real browser. For `react-call` specifically that cost is negligible — the library ships no CSS, attaches no DOM event listeners of its own (consumers' components do), and the surface under test is "did `render` show this component, did `click` trigger the consumer's `call.end()`, did the promise resolve". happy-dom handles all of that. The class of bugs a real browser would catch but happy-dom would not (e.g., a CSS regression in the demo) is not the class of bugs this test suite is responsible for; the demo at react-call.desko.dev is the real-browser validation surface for visual concerns.

## Considered options

- **Keep `packages/tests/` separate; switch only the runner from vitest-browser-react to happy-dom inside it.** Would fix the speed and CI-binary concerns but not the broken watch loop — tests would still resolve `react-call` through `dist/` and still require a rebuild on lib edits. Rejected as a half-measure.
- **Hybrid: unit-style tests colocated with happy-dom, integration/e2e tests stay in `packages/tests/` with real browser.** Adds a per-test decision ("is this a unit or integration?") that almost always falls on the same side (unit), and maintains two test infrastructures for one library's worth of meaningful surface. The "real browser as a safety net" framing it depends on would only justify itself if we had a body of tests that genuinely needed it; we don't. Rejected.
- **Full real-browser suite via `@testing-library/react-native` + Metro bundler.** Maximum fidelity for both web and React Native, but a multi-day setup with ongoing maintenance of an RN/Expo dependency chain that drifts faster than this library does. The React Native compatibility validation strategy (ADR-pending, decided in the Q5 grilling) is a static dist-lint guard scanning the published bundle for forbidden symbols — that single test catches the class of mistakes that would actually break RN consumers, without the cost of maintaining a real RN test harness. Rejected as disproportionate to the library's surface.
- **Skip migration, keep the broken watch.** Rejected because the broken watch was the user's most-cited tooling pain point in the refactor brief; leaving it would defeat the point of the overhaul phase.

## Consequences

- The library's `devDependencies` grow to include `vitest`, `@testing-library/react`, `@testing-library/user-event`, `@testing-library/jest-dom`, `happy-dom`, `react-dom`, and `@types/react-dom`. None of these ship in the publishable artifact (`files: ["dist"]` keeps `src/__tests__/` out of the tarball, and the build excludes test files from bundling). They are noise in the package.json that does not represent runtime concerns of the library.
- `react-dom` is required by `@testing-library/react` to render to a real (happy-dom) DOM. The library itself does not depend on it and stays free of DOM-specific imports in its source — the React Native compatibility surface is unchanged by this addition.
- External-mutation calls in tests (`Confirm.call(...)`, `Confirm.upsert(...)`, `Confirm.end(...)`, `Confirm.update(...)`) are wrapped in a small `withAct` helper in `shared/act.ts`: `withAct(fn)` runs `fn` inside React's synchronous `act()` so the resulting re-render flushes inside React's act-tracking and no "not wrapped in act" warning appears on stderr. The helper uses the *synchronous* overload of `act` (not `act(async () => {...})`) because the async variant deadlocked test runs when the wrapped fn returned a pending Promise (`Confirm.call(...)` returns one that doesn't resolve until the user clicks the dialog, and React 19's async-act path interacts badly with that). Sync `act()` flushes React updates synchronously and returns the wrapped fn's value unchanged, which the test can capture as a Promise to await later.
- Tests that assert dialog *absence* after an external `Confirm.end(...)` still need `await waitFor(() => expect(...).not.toBeInTheDocument())` because the lib's `createEnd` schedules removal via `setTimeout(unmountingDelay)` (default 0), and `act()` does not drain the JS macrotask queue — only React updates flush. Tests that assert absence after a `user.click(...)` do not need `waitFor` because userEvent's internal microtask draining covers the same gap. The split is documented as a follow-up trap for anyone adding new tests against `Confirm.end(...)` shapes.
- The `tsc -b` solution graph (`tsconfig.json` at the root) loses its `packages/tests` reference. The graph is now two projects: `packages/react-call` (composite) and `sites/demo` (noEmit, references react-call). Cross-workspace IDE features continue to work because react-call's `.d.ts` source is what other workspaces resolve through — tests resolve through relative imports of source files, which is direct.
- Vitest config is a per-package `vitest.config.ts` in `packages/react-call/`, not a root config with `projects` like ffdev uses. Per ADR-0005's split (`pnpm dev` against source; `dist-lint` against the published artifact), the upcoming `dist-lint` phase will need its own test project. When it lands, the per-package config can either grow to add it or be promoted to a root config with `projects`; the decision happens then, not now.
