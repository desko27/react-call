# `react-call/vite` auto-injects `Callable.displayName` so consumers don't write it

The library ships a Vite plugin exposed via the `react-call/vite` subpath export. Enabled from a consumer's `vite.config.ts` with `plugins: [reactCall()]`, the plugin walks each TS/TSX/JS/JSX module Vite serves in dev mode and, for every top-level `(export)? const X = createCallable(...)` it finds (with `createCallable` imported from `'react-call'`, optionally renamed), appends `X.displayName = 'X';` at the end of the module. The result is that the source the consumer writes shrinks back to the natural form `export const Confirm = createCallable((props) => {…})` — the two-line pattern from ADR-0010 stays as the bundler-agnostic baseline, and the plugin is opt-in sugar for Vite users who want zero ceremony.

We chose this shape because ADR-0010's `Callable.displayName = 'Confirm'` line, while idiomatic, is still ceremony — the name is typed twice and the line is easy to forget on a new callable. A bundler plugin was already pre-committed in ADR-0010 as the next step in the post-hoc + sugar plan, and a Vite plugin specifically because Vite is where `apps/vite/` already lives, where the HMR repro is canonical, and where most modern React consumers are. The plugin is gated on `config.command === 'serve' || config.mode === 'development'`, mirroring ADR-0011's principle that the registry lookup is meaningful only when HMR can re-evaluate the consumer's module; in any production build the plugin is a no-op and the consumer's bundle is byte-identical to one built without the plugin.

## Considered options

- **Separate package `vite-plugin-react-call` or `@react-call/vite-plugin`.** Standard ecosystem convention for plugins. Rejected because it splits versioning (plugin and lib must stay in lockstep — the injected line encodes the lib's setter contract from ADR-0010), doubles the release pipeline, and forces a second `pnpm add`. Co-locating under `react-call/vite` keeps one install, one version, one repo.
- **Always inject (dev + prod).** Simpler model (source maps 1:1 with runtime), but the line is dead code in production after the consumer's bundler dead-code-eliminates the dev-only setter inside the library (ADR-0011). Charging consumers ~bytes-per-callable in their prod bundle for a no-op contradicts the whole reason ADR-0011 exists.
- **Always inject, wrapped in `if (import.meta.env.DEV)`.** Tree-shakes in prod, but emits Vite-specific code into modules the consumer might inspect, and assumes a particular substitution behaviour. Cleaner to just skip emission entirely in prod builds.
- **Always inject and let the manual line win via source order.** Rejected for a subtle bug: ADR-0010's setter has first-set-wins semantics, so an injected line emitted right after `createCallable(...)` would beat a manual line that follows it, silently overriding the dev's intent. Skip-on-manual is the only safe option that preserves the manual override path.
- **Warn instead of skip on manual displayName.** Less surprising for new users but noisier for the common case (lib users intentionally setting a custom name to break a registry collision). Skip-with-no-warning matches the project's existing low-ceremony tone.
- **Permissive AST (namespace import, nested, default export).** Covers more patterns at the cost of more parsing logic and more edge cases. Namespace imports (`import * as RC from 'react-call'`) and nested-in-function declarations are rare; the strict + rename detector covers the ~95 % case and the fallback for everything else is the manual `Callable.displayName = …` line — no regression.
- **Babel plugin instead of (or in addition to) a Vite plugin.** A babel plugin would work in any toolchain that runs babel (Webpack, Metro, etc.). Deferred, not rejected — a follow-up `react-call/babel` subpath could ship later. The Vite plugin is the right first move because Vite is where ADR-0009's HMR repro lives and where the largest user base sits.
- **Use `parseAst` from Vite (Vite 5+).** Rolldown-backed `parseAst` doesn't take a `lang` option in older Vite versions and parses TSX generics (`cc<T>(…)`) as comparison expressions. `parseSync(filename, code, { lang })` (Vite 8 / rolldown) handles TSX correctly. We standardise on `parseSync`, which pins the plugin's `vite` peer to `>=8`.

## Consequences

- **New peer dependency `vite >=8` with `peerDependenciesMeta.optional: true`.** Consumers who don't use the plugin don't get prompted to install Vite. Consumers on Vite 5–7 can still use the library; they just can't use this plugin (manual `Callable.displayName = '…'` still works everywhere).
- **New publishable artifacts: `dist/vite.{js,cjs,d.ts,d.cts}`.** The tarball snapshot from ADR-0007 grows accordingly; `dist-lint` runs across both entries. The Vite plugin runs in Node at build time, never reaches the browser, and is therefore not subject to the 1 KB `size-limit` budget (which stays scoped to `dist/main.{js,cjs}`).
- **Build pipeline gains a second lib entry.** `packages/react-call/vite.config.ts` now uses Vite's multi-entry `lib.entry` map (`{ main, vite }`) with a `fileName` function. `vite` is added to `rollupOptions.external` so it isn't bundled into the plugin output.
- **`apps/vite/` becomes the canonical plugin demo.** `apps/vite/vite.config.ts` loads `reactCall()` (resolved through a source alias for `react-call/vite`, same trick ADR-0005 already uses for the runtime). `apps/vite/src/Confirm.tsx` drops the manual `Confirm.displayName = 'Confirm'` line — its presence/absence is the proof-by-screenshot that the plugin works.
- **The plugin runs `enforce: 'pre'`.** It sees the user's source before Vite's esbuild type-stripping and before `@vitejs/plugin-react`'s Fast Refresh wrapping. The injection happens at the end of the module (string append), which survives all later transforms unchanged.
- **The plugin and the library's setter contract are versioned together.** The injected line `X.displayName = 'X'` depends on ADR-0010's `Object.defineProperty` setter being present in `createCallable`. If a future major changes the setter API (e.g., to a different property name), both the runtime and the plugin update in the same commit — no cross-package coordination needed because they live in the same package.
- **No source-map adjustment for the injection.** The plugin returns `{ code, map: null }`, telling Vite to preserve the previous source map for the unchanged lines. The appended lines have no original source position, which is fine for dev-only injection that never appears in user stack traces.
- **The dist runtime bundle is unchanged.** No new code in `dist/main.{js,cjs}`. The size-limit measurement from ADR-0011 (803 / 881 B brotli) is unaffected.
