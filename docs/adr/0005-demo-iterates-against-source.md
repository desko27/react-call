# Demo iterates against `src/`: dev decoupled from `dist/`

The `sites/demo` Vite dev server resolves `react-call` to `packages/react-call/src/main.ts` (and `react-call/package.json` to the source `package.json` for the version-read use case in `HeroSection/Header.tsx`) via `resolve.alias` entries with exact regex matches. The same aliases apply in `vite build`, not just dev. The mapping is mirrored by a `compilerOptions.paths` entry in `sites/demo/tsconfig.json` so types and runtime agree. Editing library code under `packages/react-call/src/` triggers HMR in the demo with no rebuild step — the demo is a development fixture, not a consumer.

We chose this because the alternative — having the demo consume `packages/react-call/dist/` like a real npm install would — paid the cost of a full library rebuild on every save (currently ~700ms of `vite build` plus the `vite-plugin-dts` declaration rollup, plus a demo reload that loses any in-flight Call state) and gave nothing back: consumer-fidelity validation is what `dist-lint` is for (the still-pending phase that will run `publint`, `@arethetypeswrong/cli`, and a tarball snapshot against the real `dist/`), not what local `pnpm dev` is for. Making the demo behave like a consumer locally duplicates work the CI will exhaustively cover and slows the dev loop for no benefit. The same logic applies in `vite build`: the published demo at react-call.desko.dev bundles the source for distribution, which produces an identical user experience as bundling the published dist (the demo's bundler tree-shakes either way) without imposing a "build the lib first" dependency on the demo's CI build job.

The exact-match regex form for the aliases (`/^react-call$/` and `/^react-call\/package\.json$/`) is necessary because Vite's default string-form aliases do prefix matching — an alias `'react-call'` would have matched `react-call/package.json` and produced a nonsensical resolution like `packages/react-call/src/main.ts/package.json`. Each public subpath the demo consumes is enumerated explicitly; adding a new subpath to react-call's `exports` map that the demo wants to import requires a corresponding alias entry here.

## Considered options

- **Demo consumes `dist/` like a real npm install** (paired with `vite build --watch` on the library and a full demo reload on every save). Rejected for the per-save cost: ~700ms library rebuild + ~200ms demo reload, on every keystroke when iterating. The "this matches what a consumer sees" framing it relies on conflates *development fixture* with *consumer*; the consumer's view is what `dist-lint` validates, not what `pnpm dev` provides.
- **Conditional exports (`exports.development` → `src/`).** Rejected on bundler-support grounds: in 2026 the export conditions story is still inconsistent across Vite, esbuild, Rspack, webpack, and Bun, and using them would couple our dev experience to a moving target. An explicit `resolve.alias` is local to the demo's vite.config.ts, predictable, and avoids the runtime-vs-type-resolution skew that conditional exports historically produce.
- **Vite alias only, leaving `sites/demo/tsconfig.json` untouched.** Rejected because runtime would resolve `src/` but TypeScript would still read `dist/main.d.ts` via `package.json#exports`. A new export added to `packages/react-call/src/main.ts` would resolve at runtime but fail to type-check in the demo until `dist/` was rebuilt — a confusing partial-symmetry trap. Pairing the Vite alias with `tsconfig.json#paths` keeps runtime and types in lockstep.
- **Dev server only; `vite build` still resolves to `dist/`.** Rejected for asymmetry: a contributor running `pnpm --filter demo build` locally would get different bundle behaviour than `pnpm --filter demo dev`. The one-config-for-both rule removes that class of confusion. If we later need consumer-fidelity for the deployed demo specifically, that becomes a separate `sites/demo-consumer/` workspace or a CI step that swaps the alias out.

## Consequences

- The demo does not need `packages/react-call/dist/` to run. `dist/` can be absent, stale, or actively under construction without breaking `pnpm dev` or `pnpm --filter demo build`.
- Consumer-fidelity validation is fully delegated to the upcoming `dist-lint` CI phase (`publint`, `@arethetypeswrong/cli`, tarball snapshot). Locally reproducing "what does a consumer see" is not a supported `pnpm dev` workflow; a contributor who wants that runs `pnpm build` followed by `pnpm dist-lint` against the resulting artifact (once that phase lands).
- Each public subpath the demo imports must have its own alias entry. Today: `react-call` itself and `react-call/package.json`. If react-call's `exports` map ever grows (e.g., a hypothetical `react-call/headless` entry), the alias array must grow in lockstep — otherwise the demo will fail with the same prefix-match trap that triggered the explicit regex form in the first place.
- `tsconfig.json#paths` and `vite.config.ts#resolve.alias` must stay in sync. A future change to one without the other reintroduces the partial-symmetry trap above. Keeping both stanzas adjacent in their respective files (with a comment cross-referencing this ADR) is the convention.
