# `Callable.Root` is deprecated in favour of the bare component form `<Confirm />`

Since ADR-0009 made the `createCallable()` return value a function with `{ Root, call, upsert, end, update }` Object.assigned onto it (`Confirm.Root === Confirm`), both `<Confirm.Root />` and `<Confirm />` mount the same component. This ADR closes the question of which form is canonical: `<Confirm />` is the documented form going forward, `Callable.Root` is marked `@deprecated` in the public type (no removal date), and the library's own README, demos, tests, and inline comments migrate to the bare form so the lib eats its own dog food.

We chose this because keeping both forms equally first-class indefinitely is the worst of both worlds — every reader has to learn two forms for the same thing, and new code drifts to whichever happens to appear first in whatever example they read. ADR-0009 left the choice permissive ("documentation can keep the `.Root` style if backwards-look-and-feel matters"), and a verbal grilling round on the README opted to stay on `.Root` for stability; this ADR explicitly reverses that verbal decision now that more time has passed since the 1.x cut and there's no ambiguity left about whether the new shape works. The deprecation is purely soft — `Callable.Root` keeps working forever, with no scheduled removal — but new code and new readers land on the shorter form by default.

## Considered options

- **Status quo (both forms supported, README documents `.Root`).** Zero churn, but every new reader has to read past the docstring caveat that says "both work, we picked one." The longer the lib's own demos use `.Root`, the harder it is to ever move the line later. Rejected.
- **Pivot docs only, no `@deprecated`.** Switch README, demos, and tests to `<Confirm />` but leave the type untouched. No IDE signal to consumers. Halfway between status quo and real deprecation: future readers might still pick `.Root` from old tutorials, and nothing in their editor tells them otherwise. Rejected as too quiet to be useful.
- **`@deprecated` with a 3.0 removal date.** Strongest signal. Rejected because the lib has no other reason to plan a 3.0 yet; committing the removal pre-emptively binds a future major to coordinate around this one cosmetic change. The soft deprecation captures the preference without the timeline overhead.
- **`@deprecated` with "removed in a future major" (no version pinned).** More flexible than the 3.0 commitment but still promises removal. Rejected: there's no current plan to remove. If the situation changes later, escalating from "no removal date" to "future major" is just an updated JSDoc.
- **Hard removal in 2.0.** ADR-0003 already documents the 2.0 breaking scope; adding this to it would be a single-line change in the type. Rejected because the deprecation is cosmetic — every existing 1.x consumer writes `<Confirm.Root />` and would have to migrate for no functional gain. Soft deprecation lets them migrate at their own pace.

## Consequences

- **Public type `Callable<>`'s `Root` property carries an `@deprecated` JSDoc tag.** Consumers' IDEs render the property with a strikethrough on access. No type error, no runtime change. Removing the JSDoc tag is a one-line revert if we ever want to walk this back.
- **The lib's own usage migrates to `<Confirm />`.** README examples, `sites/demo`, `apps/vite`, and ~30 test files perform a mechanical `<X.Root />` → `<X />` substitution. Same component, same behaviour — the substitution is purely cosmetic. The diff is large but trivially reviewable.
- **The type test that asserts `Callable.Root` is typed as `FunctionComponent<RootProps>` stays.** It documents that the deprecated alias remains a typed-correctly part of the public API. A comment now points at this ADR so a future cleanup pass doesn't delete the test thinking it covers dead code.
- **Internal lib comments that referenced `<Confirm.Root />` as the mounting form** (in `createCallable/index.tsx` and `vite/index.ts`) are updated to `<Confirm />`. Mechanical.
- **This ADR explicitly reverses a previous verbal decision** to keep `.Root` as the canonical form in the README. The reversal is recorded here because, without it, the commit history shows the README content flipping with no in-tree explanation of why.
- **No bundle-size impact.** The `Root` property is still assigned in `Object.assign(Root, { Root, ... })` — removing it would be a runtime breaking change reserved for a future major. The JSDoc tag itself isn't shipped.
- **No README "Hot reload (HMR)" section regression.** The HMR snippet added in ADR-0010 and ADR-0012 already uses `<Confirm />` implicitly (they show only the callable declaration, not the mount). Migrating the rest of the README aligns it with that section.
