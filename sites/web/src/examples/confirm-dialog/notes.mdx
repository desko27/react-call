## When to use this

A boolean response is the right model when **the user's answer is the whole
point**. Delete confirmations, "discard unsaved changes?", "do you accept?"
— the caller doesn't need a payload back, just a yes-or-no.

If the dialog has fields, return the form data directly (see [Prompt for
input](/examples/prompt-input) or [Save form](/examples/save-form)) instead
of a boolean plus a side-channel.

## Gotchas

- **Don't pre-confirm in your handler.** A common mistake is to do
  `if (!safe) await Confirm.call(…)`. If `safe`, you skip the dialog. But
  then the call is only sometimes async — refactoring tools and reviewers
  miss the implicit branching. Make the call unconditional when it fits;
  the dialog itself can render different copy based on props.

- **The "No" path is `false`, not throw.** `call.end(false)` resolves the
  promise; rejecting the promise is reserved for unexpected failures. Cancel
  is a value, not an error.
