## Plain-data props, not JSX

It can be tempting to make a generic `<Drawer>` Callable that accepts
`body: ReactNode` and renders whatever JSX the caller hands it. **Don't.**
`.call()` is imperative — the JSX you pass is captured as a prop snapshot
at call time. The drawer won't react to state changes in the caller, and
the typed contract that makes Callables worth using collapses into
"pass me anything."

A Callable's strength is being dedicated to one use case with a clear
shape: typed props in, typed Response out. Here, `Props = { initial:
Settings }` and `Response = Settings | null` — Save resolves with the new
values, Cancel/Escape/backdrop-click resolves with `null`. The drawer
owns its working state with `useState(props.initial)`: you read the
starting values in once and report the result out once, because the call
has no reactive link back to its caller's render.

## When to reach for it

- Settings panels, filter pickers, detail views — anywhere a temporary
  side surface owns a form and reports the result back.
- Cases where the user might commit (Save) or back out (Cancel) and the
  caller needs to know which happened.
